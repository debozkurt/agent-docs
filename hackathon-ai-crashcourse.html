Tokenization: The Currency of AI

How Text Becomes Tokens

// A token is NOT a word. It can be:
// - a whole word    - part of a word
// - punctuation     - a space + word

// "Tokenization is surprisingly tricky!"
["Token", "ization", " is",
 " surprising", "ly",
 " tricky", "!"]  // 7 tokens

// Common suffixes (-ing, -tion, -ly) = 1 token
// Rare words get split into pieces:
"ChatGPT"  → ["Chat", "G", "PT"]  // 3 tokens
"🏠"       → ["🏠"]             // 1 token
"$450,000" → ["$", "450", ",", "000"] // 4 tokens

Why You Should Care

  • Cost — you pay per token, both input and output
  • Limits — context window is measured in tokens
  • Speed — more tokens = slower response
  • Multilingual — non-English text uses more tokens per word

The Cost Reality

100 API calls with Claude Sonnet costs ~$0.30. Your entire hackathon's API usage will probably cost less than lunch. Don't let cost anxiety slow you down.

The API Call: It's Just HTTP

Strip away every framework and abstraction.
This is all it is.

import anthropic                              # pip install anthropic

client = anthropic.Anthropic()                 # uses ANTHROPIC_API_KEY env var

response = client.messages.create(             # send request → get response
    model='claude-sonnet-4-20250514',
    max_tokens=1024,
    messages=[
        {'role': 'user', 'content': 'What is machine learning?'}
    ]
)

print(response.content[0].text)               # extract text from response

Five lines of real code. Everything else — LangChain, LlamaIndex, frameworks — is a wrapper around this.

Why Each Section Matters

Role

Sets expertise level and voice. "Senior backend engineer" produces different code than "helpful assistant."

Context

This can be your stack, your data, your user — whatever your code injects at runtime.

Task

One clear instruction. Ambiguity here is the #1 cause of bad outputs.

Constraints

Prevents over-engineering and scope creep. The model will add features you didn't ask for without these.

Format

Controls output shape. "Only code, no explanation" eliminates preamble you'd have to strip.

How the API Call Works

The left card shows how these sections map to the actual API call. Both are sent together on every request:

System message — the model's job description. The template stays the same, but your code injects variables at runtime — current date, user preferences, retrieved documents, etc. LLMs are stateless, so you re-send it on every call.

User message — the work ticket. This is the only part that changes per turn.

This split is a guideline. If context varies per request (e.g., different data), move it to the user message.

When RAG Works

  • Q&A over documents, knowledge bases, product data
  • Data changes frequently (no retraining needed)
  • You need the model to cite sources

When RAG Fails

  • Bad chunking retrieves the wrong passage
  • Answer sounds right but pulls from irrelevant context
  • RAG quality is an Information Retrieval (IR) problem, not an LLM problem

RAG quality is a search problem, not an AI problem. Bad answers? Fix your retrieval first.

Memory: Making AI Remember

LLMs are stateless. Every API call starts fresh.
If you want a conversation, YOU send the history.

1. Buffer 2. Summary 3. Vector Recall 4. Structured

1. Conversation Buffer

Send the full chat history every call. The list grows each turn.

1. You maintain a list of all messages
2. Each call sends the entire list to the AI
3. AI responds based on the full conversation
4. You add both the new message and response to the list
5. List grows every turn — AI sees everything

Simple. Start here. Works until you hit context limits.

2. Summary Memory

When history gets long, ask the AI to compress it.

1. History gets too long for the context window
2. Ask the AI to summarize older messages
3. Cache the summary (a short text string)
4. Future calls send: summary + recent messages
5. Re-summarize periodically as conversation grows

Trades detail for token savings. Good for longer sessions.

3. Vector Recall

Don't send everything — retrieve only the relevant past turns.

1. Embed every past turn and store as vectors
2. When a new question arrives, embed it too
3. Find the most similar past turns (cosine similarity)
4. Send only those relevant turns + the new question
5. AI responds with context it actually needs

Same cosine math as RAG. Best for apps with long histories.

4. Structured Knowledge

Extract facts from conversations. Store in a database. Query as needed.

1. Ask the AI to extract structured facts from conversation
2. Store them in a database (preferences, goals, etc.)
3. On future calls, query the DB for relevant facts
4. Inject those facts into the prompt
5. AI responds with precise, persistent knowledge

Most complex. Best for production systems.

Tool Use: Letting AI Do Things

The model requests actions. Your code executes them.
The model never runs code directly.

1. Define the Tool

1. Give each tool a name and description
2. Define what inputs it accepts (JSON schema)
3. The model reads this to understand what's available

2. Pass Tools to the API

1. Send your tools list alongside the user message
2. Model decides if a tool is needed
3. If yes: returns tool name + inputs
4. Your code executes the actual function

How Tokenization Works

Text is broken into small units (tokens) and converted into vectors the model can process.

1. Split text into tokens
"The cat sat" → ["The", "cat", "sat"]
2. Convert tokens → IDs → vectors
Each token maps to a number, then to a high-dimensional vector
3. Position is encoded
Order matters: "cat sat" ≠ "sat cat"

Text → numbers. Everything after this is vector operations.

How Attention Works

Each token computes a relevance score against every other token, assigns weights, and blends their information.

1. "cat" compares itself to "The", "sat", "on"   → similarity scores
2. Scores become weights: [0.1, 0.8, 0.1]   → "80% attention on 'sat'"
3. New representation = weighted blend of all tokens

What matters? This is how context flows between tokens.

How Compute Works

The model transforms token representations into next-token probabilities through stacked layers.

1. Apply transformer layers
Attention + feedforward layers repeatedly refine representations
2. Build contextual understanding
Each token now encodes meaning from the entire sequence
3. Output probabilities
Final layer produces likelihoods for every possible next token

Transform meaning. "Reasoning" emerges from repeated transformations, not logic rules.

How Sampling Works

The model selects the next token from a probability distribution.

1. Start with probabilities
["sat": 0.6, "ran": 0.3, "slept": 0.1]
2. Apply sampling strategy
Greedy → pick highest  |  Temperature → add randomness  |  Top-k → limit choices
3. Choose and append
One token is selected and added to the sequence

Choose next step. Same input can produce different outputs — behavior is controlled, not fixed.

The Generation Loop

The entire pipeline runs once per token. A 500-word response is ~700 iterations of this loop.

1. New token is appended to input sequence
2. Attention + Compute + Sample runs again
3. Stop when model produces an end-of-sequence token

This is why longer responses cost more (more tokens = more compute) and why streaming works token-by-token.

1. Query

The user asks a natural language question. This is the only human input in the pipeline.

"How do I reset my password?"

This raw text gets passed to the embedding model in the next step.

2. Embed

The query is converted into a vector — same embedding process as your pre-stored docs.

"How do I reset my password?"
→ [0.39, -0.71, 0.12, ...]

Your docs were embedded the same way at ingest time. Now both are in the same vector space.

3. Search (Deterministic)

Compare the query vector against every stored doc vector using cosine similarity. Pure math — same query always returns the same results.

Query: [0.39, -0.71, 0.12, ...]

0.91 ✓ "Password reset via Settings"  [0.41, -0.72, 0.08, ...]
0.87 ✓ "Change your login credentials"  [0.38, -0.69, 0.15, ...]
0.23 ✗ "Billing FAQ and payment methods"  [-0.55, 0.33, 0.61, ...]

No LLM involved here. This is an Information Retrieval (IR) operation.

4. Augment (Deterministic)

Take the top matching chunks and inject them into the prompt as context. This is literally string concatenation.

# build the prompt
context = top_matches[:3]
prompt = f'Given: {context}\n\nAnswer: {query}'

RAG is just: stuff relevant docs into the prompt. That's the whole trick.

5. Generate (LLM)

The LLM receives the augmented prompt and generates an answer grounded in the retrieved context.

resp = client.messages.create(
    model='claude-sonnet-4-20250514',
    max_tokens=1024,
    system=context,  # retrieved docs
    messages=[{'role': 'user', ...}]
)

This is the only probabilistic step. Everything before it was deterministic math.

1. Plan

The model receives available tools + conversation history and predicts the next step. Tools can be functions, APIs — or even other agent loops (sub-agents) that run their own cycle internally.

resp = client.messages.create(
    model='claude-sonnet-4-20250514',
    max_tokens=1024,
    tools=[search_tool, calc_tool],
    messages=history
)

# resp.content contains blocks:
# TextBlock → final answer (done)
# ToolUseBlock → tool request (continue)

At its core: next-token prediction over the tool menu + context. Advanced systems add evaluators, routers, and sub-agents — but this loop is always the foundation.

2. Act

Extract the tool request from the response. Your code executes it — the model never runs code, it only generates a structured request.

# extract tool call from response
tool_block = next(b for b in resp.content
                if b.type == 'tool_use')

# your code runs the actual function
result = run_tool(
    tool_block.name,  # e.g. 'search'
    tool_block.input  # e.g. {'query': 'revenue Q3'}
)

The safety boundary: the model requests, you execute. Same pattern as single tool use.

3. Observe

Append both the assistant's tool request and the tool result to history. The model sees this on the next call — it has no live awareness. In advanced systems, evaluators (separate LLM calls) can grade the result before continuing.

# 1. record what the model requested
history.append({'role': 'assistant',
    'content': resp.content})

# 2. record the tool result
history.append({'role': 'user',
    'content': [{'type': 'tool_result',
      'tool_use_id': tool_block.id,
      'content': result}]})

# optional: evaluator grades before looping
# grade = evaluate(result)

Both messages required: the assistant's request + the tool's response. Evaluators and sub-agents plug in here as additional LLM calls or nested loops.

4. Loop

The entire agent is a while loop. If the response contains a tool_use block, keep going. If the model returns only text, it's done.

while True:
    resp = client.messages.create(
        model='...', tools=tools, messages=history)

    tool_calls = [b for b in resp.content
                 if b.type == 'tool_use']

    if not tool_calls:  # text only → done
        break

    # act
    history.append({'role': 'assistant',
        'content': resp.content})
    for tool in tool_calls:
        result = run_tool(tool.name, tool.input)
        history.append({'role': 'user',
            'content': [{'type': 'tool_result',
              'tool_use_id': tool.id,
              'content': result}]}) # observe

That's the entire agent. A while loop, tool extraction, execution, and history updates. Everything else is a wrapper around this.
